# TruchieEmu — Design Context

> Persisted design guidelines for all future TruchieEmu work. These principles should inform every UI change, component addition, and design review.

## Users

### Who They Are
- **Casual retro gamers** looking for the simplest way to relive childhood games
- Users who may not be technical but want a familiar, modern macOS experience
- People who remember the *feeling* of playing classic games more than the technical details

### Context of Use
- Relaxed, at-home use on a Mac
- Browsing their game collection, picking something to play
- Managing libraries over time (adding ROMs, organizing favorites, customizing categories)

### Job to Be Done
- "I want to find and play my retro games with zero friction"
- "I want my library to look good and feel organized"
- "I want it to just work — no config, no fuss"

## Brand Personality

### Voice & Tone
- **Intuitive** — The UI should feel obvious. Users should never wonder what to do next. Clear labels, predictable patterns, progressive disclosure where needed.
- **Modern** — Not "retro-styled." The chrome is contemporary, letting the retro content (box art, screenshots, gameplay) be the nostalgic element.
- **Useful** — Every screen earns its place. Features are there when needed, hidden when they're not.

### 3-Word Personality
**Intuitive · Modern · Useful**

### Emotional Goals
- **Nostalgia** — The games themselves are the star. The UI should fade back and let box art, screenshots, and memories take center stage.
- **Confidence** — Users should feel like they're in control, never lost or confused.
- **Delight** — Small moments of polish (smooth transitions, satisfying interactions, tasteful animations) make the app feel alive.

## Aesthetic Direction

### Visual Tone
- Consumer entertainment app feel, not a pro tool or config panel
- Think **Apple Arcade meets Playnite** — clean, spacious, media-forward
- Reference: **OpenEmu** as an anti-aspiration for simplicity; RetroArch as what *not* to look like (overwhelming, config-heavy)

### Theme
- **Both dark and light mode** with equal attention
- System theme should be respected by default
- Custom theme support is a goal

### Color
- Let box art and game imagery provide most of the color
- Use accent colors sparingly and purposefully: system colors for badges, status indicators, and interactive states
- No "favorite" signature colors — the games are the palette

### Typography
- SF Pro (system default) — clean, legible, native feeling on macOS
- Clear hierarchy: game titles prominent, metadata secondary, labels subtle

### Spacing & Layout
- Generous breathing room. Avoid cramped, dense UI
- Card-based layouts with clear visual grouping
- `NavigationSplitView` for master-detail patterns
- Grid view for games (media-first), list view for scanning

## Design Principles

### 1. Media First, Chrome Second
Box art, screenshots, and game art are the visual heroes. The UI frames them, doesn't compete with them. Use clean backgrounds, subtle shadows, and restrained borders to let artwork shine.

### 2. Progressive Disclosure
Surface only the most important controls up front. Advanced settings (shaders, core options, bezels) live behind clear navigation paths. The default experience should feel simple; depth should feel discoverable.

### 3. Predictable Patterns
Follow macOS conventions where they exist. `NavigationSplitView`, standard context menus, system sheet presentations, keyboard shortcuts. Don't reinvent navigation. Users should feel at home immediately.

### 4. Nostalgia Through Content, Not Chrome
The app itself should feel modern and clean. Retro aesthetics come from the game art, not skeuomorphic UI elements, wood textures, or pixel fonts. Let the games be the retro element.

### 5. Respect the User's Library
Treat the game library like a curated collection. Beautiful grid/list views, smart filtering, clean organization. This shouldn't feel like a file manager — it should feel like a personal gaming shelf.

## Key Technical Context

### Tech Stack
- **SwiftUI** for all views (macOS native)
- **Metal** shaders for CRT/display effects
- **libretro** cores for emulation (loaded via bridging)
- **SQLite** for persistence (ROMs, metadata, settings)
- `NavigationSplitView` for main app layout

### Current Component Patterns
- `ModernSectionCard` — dark-themed card component for Game Detail view
- `MetadataRow` — aligned label/value pairs with optional copy action
- `FilterChipView` — horizontal scrollable filter chips with active color states
- `GameCardView` — grid item with box art, title, category badges
- `GameListRowView` — list row with thumbnail, metadata, playtime stats
- `CategoryBadgeView` — small colored pill for category tags

### Design Debt to Address
- Dark mode detail view has hardcoded `.white.opacity(x)` colors — not light-mode compatible
- No design token system (colors, spacing, radii are scattered)
- Some views mix layout concerns (padding, spacing) with presentation
