# Design Document: High Framerate (60 FPS) Subsystem

## 1. Objective

To provide a configurable way to run games—specifically those originally locked at 15, 20, or 30 FPS (like many N64 and PS1 titles)—at a smooth **60 FPS**.

**Crucial Distinction:**

* **Emulator Speed:** Emulators always *output* at 60Hz (for NTSC), but the game engine inside might only update the image every 2 or 3 frames.
* **High Framerate (HFR):** Modifying the game's internal logic or the emulator's clock to force the game engine to render unique motion every single frame.

---

## 2. Implementation Methods

There is no "magic button" in the Libretro API to force 60 FPS globally because every game handles timing differently. Your frontend must implement a **three-tiered approach**:

### Tier A: Core-Specific Overclocking (CPU Speedup)

Some games (like *Star Fox* or *GoldenEye 002*) lag because the original hardware was too slow. By "overclocking" the emulated CPU, the game naturally hits its target frame rate more consistently.

* **Logic:** Expose specific Core Options to the user.
* **Key Variables:**
  * **PS1 (SwanStation):** `CPU Overclock` (100% to 500%).
  * **N64 (Mupen64Plus-Next):** `CountPerOp` (Changing this from 2 to 1 effectively doubles the emulated CPU cycles).
* **Frontend Task:** Map these specific core options to a "Performance" toggle in your UI.

### Tier B: Internal 60 FPS Cheats (The "Patch" Method)

Many 30 FPS games (like *Ocarina of Time* or *Bloodborne*-style PS1 demakes) have specific memory addresses that control the "Frame Limiter."

* **Logic:** Use your **Cheat Subsystem (from Design Doc #8)** to automatically apply 60 FPS patches.
* **Database:** You should bundle or download the [60FPS Master Patches](https://github.com/mapi-84/60fps-patches) for N64 and PS1.
* **Workflow:**
    1. Detect game.
    2. Check if a `60fps.cht` exists for this TitleID.
    3. Auto-apply the cheat when the game starts.

### Tier C: Motion Interpolation / VRR (Frontend Side)

If the game cannot be patched, you can improve the *perception* of 60 FPS using your **Shader Subsystem (from Design Doc #3)**.

* **Black Frame Insertion (BFI):** If the user has a 120Hz monitor, inserting a black frame between game frames reduces motion blur, making 30 FPS look significantly crisper.
* **LCD Ghosting Shaders:** Simulates the slight blur of high-end displays to mask "judder" in low-framerate games.

---

## 3. UI/UX: The "Boost Mode" Feature

Since 60 FPS can break some games (making them run at 2x speed or breaking physics), you should implement a **"Boost Mode"** overlay menu:

1. **Toggle: "Force 60 FPS"**
    * If ON: Activates the CPU Overclocking variables and searches for a 60FPS cheat code.
2. **Toggle: "Synchronous / Asynchronous Compute"**
    * Allows the GPU to render as fast as possible regardless of the CPU's timing (reduces input lag).
3. **Warning Notification:**
    * Show a toast message: *"60 FPS Boost enabled. Note: This may cause game physics to run faster than intended."*

---

## 4. Handling Game Physics (The "Game Speed" Problem)

Many retro games tie their physics to the framerate. If you force *Sonic* to 60 FPS, he might run twice as fast.

**Frontend Solution:**

* Implement a **Frame Rate Target** vs **Game Speed** ratio.
* If the core supports it (like `DuckStation/SwanStation`), enable **"GTE Overclock"** and **"Widescreen Hack"** alongside FPS boosts. This ensures the 3D geometry is updated at 60Hz while keeping the game logic at its original speed.

---

## 5. Technical Implementation Checklist

### [ ] VRR / G-Sync Support

Ensure your Metal/OpenGL/Vulkan swapchain is configured for **Variable Refresh Rate**.

* In your window creation code, set `swap_interval = 0` if VRR is detected. This allows the frontend to display whatever frames the core produces immediately.

### [ ] Core Option Presets

Create a "Turbo" profile for specific cores:

* **N64:** Set `Parallel-RDP` to high and `CountPerOp` to 1.
* **PS1:** Set `Internal Resolution` to 2x and `CPU Overclock` to 150%.

### [ ] Cheat Integration

In your Cheat Parser, add a "Metadata Tag" for 60FPS:

```ini
# Example in Super Mario 64.cht
cheat0_desc = "60 FPS (HACK)"
cheat0_code = "81000000 0000"
cheat0_is_fps_hack = true
```

Your UI can then show a dedicated "60 FPS Boost" button that simply toggles this specific cheat.

---

## 6. Summary for the User

In your frontend documentation, explain to users:
> "To achieve 60 FPS in N64 and PS1 games, our frontend uses a combination of **CPU Overclocking** and **Memory Patching**. If a game feels like it is moving in 'fast forward,' disable the 60 FPS boost, as that specific game's physics are tied to its framerate."
